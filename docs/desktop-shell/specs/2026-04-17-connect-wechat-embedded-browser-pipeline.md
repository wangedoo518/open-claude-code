## 一、核心洞察

### Replit 部署流程的 UX 精髓（截图分析）

Replit 的 Publishing 流程有三个值得学习的模式：

| 模式 | Replit 实现 | 价值 |
|------|-----------|------|
| **顶部进度条** | Provision → Security Scan → Build → Bundle → Promote，绿色依次点亮 | 用户始终知道"到哪一步了" |
| **可展开日志** | "View logs" / "Hide logs" 切换，日志实时滚动 | 想看细节可以看，不想看不打扰 |
| **完成弹窗** | "Congrats! Use the link below to share" + 复制链接 | 清晰的成功信号 + 立即可用的结果 |

### Obsidian Custom Frames 的嵌入模式

关键技术发现：
- **Electron/Tauri 桌面端用 webview**，有完整浏览器能力（Cookie、登录、表单）
- **ClaudeWiki 已有 MinApp 基础设施**（`minapps.ts` + `webviewStateManager.ts`），可直接复用
- **webview 可以嵌入 Cloudflare Dashboard 和微信客服后台**，用户在 App 内完成所有操作

### 关键改进：从"弹窗"到"嵌入式工作台"

v1 方案（纯 Modal）的问题：
- Pipeline 执行时用户只能看到进度条 + 日志，无法看到浏览器操作过程
- Cloudflare 注册、微信授权需要跳转到外部浏览器，打断用户流程
- 用户不知道"自动化在做什么"

v2 方案（嵌入式浏览器）的改进：
- **左侧 Pipeline 进度 + 右侧嵌入浏览器**，用户实时看到自动化操作
- Cloudflare 注册/部署 → 在 App 内嵌的 webview 中完成
- 微信客服授权 → 在 App 内嵌的 webview 中扫码
- 全程不离开 ClaudeWiki

## 三、用户旅程（纯一键接入，无手动配置）

> **设计决策**：去掉"手动配置"选项。原因：
> 1. 手动配置需要用户理解 corpid/secret/AES key，门槛太高
> 2. 一键接入覆盖了手动配置的全部场景
> 3. 减少用户选择 = 减少流失

```
Step 0: 侧边栏点击"微信助手 · 未连接"
    │
    ▼
Step 1: 确认开始页（不是选择页）
    │   标题: "一键接入微信客服"
    │   描述: 自动完成全部配置，你只需要完成 2 次手动操作
    │   [开始接入 →] 按钮 → navigate("/connect-wechat")
    │
    ▼
Step 2: 嵌入式 Pipeline 页面（全屏双栏）
    │
    ├── Phase ①: Cloudflare 账号注册
    │   └── 右侧 webview 打开 dash.cloudflare.com/sign-up
    │       ⚠ 用户需手动操作：完成真人验证（CAPTCHA/Turnstile）
    │       左侧提示："请在右侧完成 Cloudflare 注册和真人验证"
    │       Pipeline 轮询检测注册完成 → 自动进入下一步
    │
    ├── Phase ②: 部署中继服务器
    │   └── 自动执行，右侧 webview 显示部署过程
    │       用户无需操作，观看即可
    │
    ├── Phase ③: 微信客服授权
    │   └── 右侧 webview 打开 kf.weixin.qq.com
    │       ⚠ 用户需手动操作：用手机微信扫码
    │       左侧提示："请用手机微信扫描右侧二维码"
    │       Pipeline 轮询检测授权完成 → 自动进入下一步
    │
    ├── Phase ④: 回调 URL 配置
    │   └── 自动执行，用户无需操作
    │
    └── Phase ⑤: 创建客服账号
        └── 自动执行，用户无需操作
    │
    ▼
Step 3: 完成页（Replit Congrats 风格）
    ├── 🎉 微信客服接入完成！
    ├── 客服二维码 + 复制链接
    ├── 使用方法提示
    └── [完成，回到主页 →]
```

**用户只需 2 次手动操作**：
1. Cloudflare 真人验证（CAPTCHA）
2. 微信扫码授权

其余全部自动化。

#### 关键交互：浏览器抽屉自动展开/收起

右侧嵌入浏览器是**抽屉式面板**，根据当前步骤自动展开或收起：

| Phase | 用户操作 | 浏览器抽屉 | 左侧占宽 |
|-------|---------|----------|---------|
| ① CF 账号 | **手动：真人验证** | **展开** → dash.cloudflare.com | 左 340px + 右 flex-1 |
| ② 部署中继 | 自动 | **收起** | 左 全屏 |
| ③ 微信授权 | **手动：扫码** | **展开** → kf.weixin.qq.com | 左 340px + 右 flex-1 |
| ④ 回调配置 | 自动 | **收起** | 左 全屏 |
| ⑤ 创建账号 | 自动 | **收起** | 左 全屏 |

**抽屉行为**：
- 展开/收起有 300ms 水平滑动动画（`transition: width 300ms`）
- 用户也可以手动点击「◀ 收起浏览器」/「▶ 展开浏览器」切换
- 自动步骤时抽屉收起，Pipeline 面板居中显示，体验更聚焦
- 手动步骤时抽屉自动展开，用户注意力被引导到右侧浏览器

**视觉规则**：
- 需要手动操作的步骤：⚠ 黄色提示框 + 浏览器抽屉自动展开
- 自动执行的步骤：✅ 绿色 "无需操作" + 浏览器抽屉收起 + 居中进度动画

## 五、技术方案

### 5.1 webview 嵌入（复用 MinApp 基础设施）

ClaudeWiki 已有 MinApp webview 机制（`minapps.ts` + `webviewStateManager.ts`），
Pipeline 页面的右侧浏览器直接复用此机制：

```typescript
// Pipeline webview 不注册为 MinApp，而是作为 PipelinePage 的内部组件
// 使用 Tauri 的 <webview> 或降级为 <iframe>

interface PipelineWebviewProps {
  url: string;           // 当前阶段要加载的 URL
  onUrlChange?: (url: string) => void;  // URL 变化回调
}

function PipelineWebview({ url }: PipelineWebviewProps) {
  return (
    <div className="flex flex-col h-full">
      {/* 导航栏 */}
      <div className="flex items-center h-9 border-b px-2 gap-1">
        <button>◀</button>
        <button>▶</button>
        <button>🔄</button>
        <span className="flex-1 truncate text-xs">{url}</span>
        <button>🔗</button>
      </div>
      {/* webview/iframe */}
      <webview src={url} className="flex-1" />
    </div>
  );
}
```

### 5.2 路由设计

Pipeline 页面是一个**新路由**，不是 Modal：

```typescript
// ClawWikiShell.tsx 新增路由
<Route path="/connect-wechat" element={<ConnectWeChatPipelinePage />} />
```

流程：
1. 用户点击侧边栏"微信助手 · 未连接"→ 弹 ConnectWeChatModal（确认开始页）
2. 用户点击"开始接入"→ Modal 关闭 → `navigate("/connect-wechat")`
3. Pipeline 页面全屏渲染，左侧进度 + 右侧 webview
4. Phase ① 等待用户在 webview 中完成 Cloudflare 真人验证
5. Phase ②③④⑤ 自动/扫码混合执行
6. 完成后显示 Congrats 页面
7. 点击"完成"→ `navigate("/ask")`，侧边栏状态变绿

### 5.3 组件拆分

| 组件 | 职责 |
|------|------|
| `ConnectWeChatPipelinePage.tsx` | 全屏路由页面，管理 Pipeline 状态机 |
| `PipelineProgressBar.tsx` | 顶部 5 步进度条（Replit 风格） |
| `PipelineStepPanel.tsx` | 左侧面板：当前步骤描述 + 已完成列表 + 日志 |
| `PipelineWebview.tsx` | 右侧嵌入浏览器（webview/iframe） |
| `PipelineCongratsPage.tsx` | 完成页 |

### 5.4 Tauri webview vs iframe

| 环境 | 方式 | 能力 |
|------|------|------|
| **Tauri 桌面端** | `<webview>` 标签 | 完整浏览器能力，Cookie 持久化，登录态保持 |
| **浏览器开发模式** | `<iframe>` 降级 | 受限于 X-Frame-Options，Cloudflare/微信可能阻止 |

**降级策略**：如果 iframe 被目标网站阻止，显示提示 + "在浏览器打开" 按钮，
让用户在外部浏览器完成该步骤，Pipeline 侧通过轮询 API 检测完成状态。

| 维度 | v1（纯 Modal） | v2（嵌入式浏览器） |

| **用户可见性** | 只看到进度条 + 文字日志 | 实时看到浏览器操作过程 |
| **Cloudflare 注册** | 跳转外部浏览器 | App 内 webview 完成 |
| **微信授权扫码** | Modal 内显示 QR 图片 | webview 直接显示微信后台 |
| **空间利用** | 小 Modal 窗口 | 全屏双栏布局 |
| **信任感** | "后台在做什么？" | "我看到了每一步" |
| **技术复杂度** | 低（纯 React） | 中（需要 webview 集成） |
| **降级能力** | — | iframe 被阻止时降级为外部浏览器 |
